[Will s2 — 1:45]
"Scenario 1: Duplicate Account Creation.

Sally Sparrow tries to create an account. But here's the thing—her brother Sullivan already has an account with the email sparrow@example.com. So when Sally attempts to create an account with the same email, the system rejects her.

What happens behind the scenes? UserList maintains the users via email address. When we try to add Sally's duplicate entry, the HashMap's containsKey check fails, and addUser returns false. The UI then prints 'Sorry, we couldn't create the user. Email already exists.'

The takeaway here is that the rejection isn't about family relationships—it's purely email-based duplicate detection. That's what makes the validation rule deterministic and testable. No fuzzy logic, no name matching. Just does this email already exist in our JSON? Yes? Reject.

This demonstrates the first layer of data integrity: preventing duplicate records at insertion time."

[Daruis S1 — 2:00]
"Scenario 2: Successful Account Creation and Login.

Sally realizes the issue, so she changes her email to sally@example.com. This time when she tries to create the account, the email is unique, so the system accepts it.

Here's the critical part: her password gets hashed using BCrypt before it's stored in memory and persisted to JSON. BCrypt gives us a salted hash, so even if someone accessed our JSON file, they can't just read passwords in plaintext. Each password has a unique salt, so even identical passwords produce different hashes.

When Sally logs in with her credentials, SystemFacade calls login on UserList, which uses BCrypt.checkpw() to verify her password against the stored hash. If it checks out, her user object is loaded, and she's authenticated. She also gets a currentUser session object so the facade knows who's acting right now.

Notice also that each user has a lastLogin timestamp. When Sally logs in, that gets updated. This is useful for tracking activity and user engagement over time.

So scenario 2 shows us account creation with proper password security and successful authentication."

Under the hood, DataWriter recursively serializes users, questions, sections, answers, and comments into JSON objects and arrays, then writes formatted output for readability.

When the app starts again, DataLoader parses those files and rebuilds the same nested Java objects in memory.

That is the core value of this layer: continuity across sessions. What users create in one run is still available in the next run."
